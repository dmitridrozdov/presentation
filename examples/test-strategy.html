AWB Test Strategy

Software Testing Pyramid

1. Everything should be tested (as much as possible)

2. All tasks should have acceptance criteria (can be multiple criterias)

3. After the task completed QA/Dev engineer should verify task according to acceptance criteria

4. Task life cycle


  • Is functional testing required
  • Do we need update / add to regression tests

5. Regression test strategy


  • How often should be executed
  • Do we need to have automatic execution plan
  • Reports

6. Defect management


  • Tracking
  • Fixing
  • Verifying

7. Test Development (including unit tests)


  • Plans
  • Fixing
  • Data

8. Test coverage


  • What should be covered
  • Different test types

9. As result of test strategy should be created the system


  • Unit tests
  • Functional tests
  • Integration tests
  • Regression tests
  • Reports

Questions

Question 1:

Is there any specific plan related to our components?
GUI testing, API testing and Regression testing


Answer:

  • GUI testing: Create a fully automatic set of tests (in progress)
    and execute them during the regression cycle.
  • API testing: At the moment we can verify API using swagger and tests created by Georgy.
    Later on, we can create automatic verification.
  • Regression testing: Currently we have 3 regression tests (ds.swoop, scoring pipeline and aqdq)
    and I'll add tests for Orbis metadata. We have to descide how often we are going to run these tests.

Question 2:

Does "regression tests" only refers to current regression tests for datascience.swoop/scoring pipeline, or is it in the more broader sense testing if current proposed system produces the same result as the previous system?


Answer:

Regression tests will include existing tests for datascience.swoop/scoring/aqdq pipelines plus new Orbis UI automation for scoring and aqdg pipelines

Question 3:

What is the plan for automatic execution - is it automatic in that e.g. it is that triggered from PR merge, or the current steps be automated?


Answer:

Ideally all regression tests should be triggered automatically once pull request merged to master. As per my understanding to achieve this all existing regression tests should be slightly improved and, in that case, the most important question will be time and resources.

Question 4:

Are there any plans for integration testing?


Answer:

Our system is quite complex and I'm sure that integration tests are required. Probably we have to execute them manually at this development stage.
